# 인터뷰 가이드 — 스킬러들의 수다 (Codex)

채팅-우선 질문을 쓴다. 답이 생성될 결과물을 바꿀 때만 묻는다. 모든 선택 질문은 `shared/questioning-policy.md §A`의 번호형 블록으로 렌더한다 — Codex에는 `question prompt` 카드가 없다.

## 1. 4 관점 분석 설계

검증 다중성을 위해 **인원수 4는 유지**하되, 4명의 정의는 매번 아이디어 도메인을 보고 동적 생성한다(`references/personas.md`). 기본은 메인 스레드에서 4 관점을 직접 종합하고, 사용자가 명시적으로 요청하고 환경이 지원할 때만 bounded sub-agent로 위임한다.

### 기획자 관점
- **역할**: 방향 설정, 범위 정의, MVP 판단
- **핵심 질문**: "이 스킬이 정말 필요한가? 누가 쓰는가?"
- **분석 항목**: 목적, 대상 사용자, 범위(MVP vs 확장), 트리거 키워드, 기획 우려사항

### 사용자 관점
- **역할**: 실제 사용 시나리오 검증, UX 평가
- **핵심 질문**: "나라면 이걸 어떻게 쓸까? 불편한 점은?"
- **분석 항목**: 사용 시나리오, 입력 방식, 출력 기대, 사용성, 실망 포인트

### 전문가 관점
- **역할**: 기술 실현 가능성 판단, 워크플로우 설계
- **핵심 질문**: "이 분야는 이런 점을 조심해야 해"
- **분석 항목**: 워크플로우 제안, 단계 타입, 필요 도구, 컴포넌트 타입, 기술 리스크

### 검수자 관점
- **역할**: 엣지 케이스, 실패 시나리오 검증
- **핵심 질문**: "이거 이 경우에도 돼? 실패하면?"
- **분석 항목**: 엣지 케이스, 에러 핸들링, 입력 검증, 테스트 시나리오, 실패 복구

## 2. 분석 항목 템플릿

메인 스레드에서 각 관점을 적용할 때, 또는 sub-agent에 위임할 때 아래 항목으로 분석한다. `{아이디어}`를 사용자 아이디어로, `{추가정보}`를 Phase A 추가 정보로 교체한다.

### 기획자 분석
```
아이디어: {아이디어}
{추가정보}

1. 목적: 이 스킬이 해결하는 핵심 문제
2. 대상: 누가 쓸 건지 (나만/팀/불특정 다수)
3. 범위: MVP로 먼저 만들 핵심 기능 vs 나중에 추가할 기능
4. 트리거 키워드: 자연스러운 키워드 5개 (한/영 혼합)
5. 우려사항: 기획 관점 주의점
→ "기획자 한줄 요약: {요약}"
```

### 사용자 분석
```
1. 사용 시나리오: 실제로 어떤 상황에서 쓸지
2. 입력 방식: 사용자가 뭘 제공하는지 (텍스트/파일/URL), 가장 편한 방식
3. 출력 기대: 결과물 형태 (텍스트 요약/파일/보고서)
4. 사용성: 처음 쓰는 사람도 이해하는지, 불편할 수 있는 점
5. 실망 포인트: 기대에 못 미칠 수 있는 부분
→ "사용자 한줄 요약: {요약}"
```

### 전문가 분석
```
1. 워크플로우 제안: 단계별 실행 흐름 (3-5단계)
2. 단계 타입: 각 단계가 prompt/script/api_mcp/rag/review/generate 중 무엇인지
3. 필요한 도구: 외부 API, MCP/App, 스크립트
4. 컴포넌트 타입: 스킬/플러그인 중 적합한 것과 이유 (Codex엔 command/agent 없음)
5. 기술적 리스크: 구현 시 까다로운 부분
6. 추천 MCP/App: 유용한 외부 연동 1-3개 (references/mcp-catalog.md 참조), 왜 필요한지 + 설치 안내
→ "전문가 한줄 요약: {요약}"
```

### 검수자 분석
```
1. 엣지 케이스: 정상 경로 외 예외 상황 3개 이상
2. 에러 핸들링: 각 엣지 케이스 대응 방안
3. 입력 검증: 잘못된 입력 처리
4. 테스트 시나리오: 정상/비정상 각 2개
5. 실패 시 복구: 중간 실패 시 사용자 경험
→ "검수자 한줄 요약: {요약}"
```

### 결과 종합
- 4 관점 분석을 대화형 종합으로 변환한다.
- 의견 충돌이 있으면 **반드시** 사용자에게 노출하고, 충돌 자체를 §A 번호 블록 질문으로 만든다.
- 종합 형식: 페르소나 이름 + 한줄 요약 + 핵심 분석.

## 3. 인터뷰 흐름

```
Phase A: 아이디어 수집 (열린 질문 1-2개)
    ↓
Phase B: 4 관점 분석 → 종합 + 방향 확인 (§A 번호 블록)
    ↓
Phase C: 상세 인터뷰 (§A 번호 블록 0-2개, 필요시만)
    ↓
Phase D: 워크플로우 설계 + 확인 (§A 번호 블록)
    ↓
Phase E: 파일 생성 → Phase E-verify (quick_validate)
    ↓
Phase F: Eval 실행 + 검토 (반복)
```

### 질문 총량
- Phase A: 1-2개 (아이디어, 열린 질문)
- Phase B: 1개 (방향 확인, §A 블록)
- Phase C: 0-2개 (불확실한 것만)
- Phase D: 1개 (워크플로우 확인)
- **총 인터뷰 질문: 3-6개** (고도화 제외)

## 4. §A 번호 블록 규칙 (재확인)

- 핵심 문제 발굴(Phase A 첫 턴)은 **열린 텍스트 질문**. 분류/설정 고르기는 **번호 블록**.
- 추천안은 항상 1번 + "(추천)".
- 각 선택지에 "무엇인지 + 트레이드오프" 한 줄.
- 마지막 선택지는 "문장으로 직접 수정 요청"("Other" 대체).
- 여러 개 고를 수 있으면 "여러 개면 1,3처럼 적어주세요" 안내.
- 사용자를 번호에 가두지 않는다 — 번호든 문장이든 받는다.

## 5. 의도 파악 데이터 모델

| 필드 | 타입 | 설명 | 파악 주체 |
|------|------|------|----------|
| purpose | text | 해결하는 문제 | 기획자 관점 |
| input_type | enum | file/text/url/mixed | 사용자 관점 |
| output_type | enum | file/text/message/mixed | 사용자 관점 |
| target_user | text | 누가 쓰는지 | 기획자 관점 |
| trigger_keywords | list | 트리거 키워드 | 기획자 관점 |
| domain | text | 전문 분야 | 전문가 관점 |
| workflow | list | 단계별 흐름 | 전문가 관점 |
| constraints | list | 제약 조건 | 검수자 관점 |
| edge_cases | list | 엣지 케이스 | 검수자 관점 |
| tone | text | 말투/스타일 | Phase C (필요시) |
| reference_data | list | 참조 데이터 | Phase C (필요시) |

**핵심**: 모든 항목을 직접 묻지 않는다 — 4 관점이 먼저 분석하고 불확실한 것만 사용자에게 확인한다(§2c).

## 6. 인터뷰 실패 방지

- **아이디어가 모호하면** — Phase A에서 과거 행동 앵커로 1개 더(§2b). "지금은 그걸 어떻게 하세요?"
- **관점이 엇갈리면** — Phase B 종합 시 충돌을 노출하고 §A 블록으로 결정 요청.
- **너무 크게 요청하면** — 기획자 관점이 MVP 범위 제안. "먼저 이것부터, 나중에 확장."
- **사용자가 "모르겠어요"면** — 가장 합의된 방향을 기본값으로 제안(§2c, 과잉질문 금지).
