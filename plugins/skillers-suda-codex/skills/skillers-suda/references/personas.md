# Personas — 4 관점 동적 생성 가이드

> **인원수 4는 검증 다중성의 본질로 유지**.
> **4명의 정의는 매번 사용자 아이디어의 도메인을 보고 동적 생성**.
> 본 문서는 enum 카탈로그가 아니라 **동적 생성 절차 + 참고 예시**다.

## 본 문서 사용 원칙

- 메뉴처럼 쓰지 않는다: "카탈로그에서 도메인 하나 골라 4명 가져오기" 금지.
- 예시처럼 쓴다: "도메인별 변환 사례를 본 뒤, 아이디어에 맞게 4명을 매번 새로 정의".

## 핵심 원칙

1. **개수 4는 보존** — 검증 다중성이 SKILL.md 품질의 본질.
2. **정의는 매번 동적 생성** — 아이디어 분석 → 4명 즉석 정의.
3. **4명은 항상 다른 관점** (같은 관점 중복 금지).
4. **본 문서 예시는 메뉴가 아닌 패턴 학습용**.

## 동적 생성 절차

Phase B 진입 직전, 사용자 아이디어를 보고 4명을 생성한다.

### Step 1: 도메인 핵심 키워드 추출
- 누가 쓰는 도구? (사용자 유형)
- 어떤 결정을 도와주나? (의사결정 영역)
- 무엇이 잘못되면 큰 문제? (위험 영역)
- 어떤 외부 제약? (규제/도메인 표준)

### Step 2: 4 관점 매핑

추출된 키워드로 **항상 다음 4 차원이 모두 다른 관점**이 되도록 정의:

| 차원 | 페르소나 정의 방향 |
|---|---|
| **방향성 / 비전** | "이걸 왜 만드는가? 누가 쓰는가? 핵심 가치는?" |
| **실제 사용** | "내가 실제로 이 도구를 쓸 때 어떨까? 진입장벽은?" |
| **도메인 전문성** | 이 도메인에서 잘못되기 쉬운 점, 규제, 모범 사례 |
| **검증 / 엣지** | "이거 X 케이스에도 동작? 보안 위험? 규제 위반?" |

### Step 3: 도메인 특화 명명
각 관점을 도메인에 맞게 즉석 명명. 아래 §참고 예시처럼 도메인 키워드를 변환.

### Step 4: 사용자 confirm 후 분석
4 관점을 메인 스레드에서 종합하거나(기본), 사용자가 요청하고 환경이 지원하면 동일한 4명으로 bounded sub-agent에 위임한다.

## 참고 예시 — 동적 생성 사례 모음

> **이 예시들은 메뉴가 아니다.** 도메인이 비슷해도 세부에 따라 4명을 다시 정의한다.

### 예시 0 — 기본 (도메인 미특정 / 모호)
| 페르소나 | 핵심 질문 |
|---|---|
| 기획자 | "누가 쓸 건데? 뭘 해결하는 거야?" |
| 사용자 | "나라면 이걸 어떻게 쓸까?" |
| 전문가 | "이 분야는 이런 점을 조심해야 해" |
| 검수자 | "이거 이 경우에도 돼?" |

### 예시 1 — 헬스케어
| 페르소나 | 핵심 질문 |
|---|---|
| 의사 | "임상적으로 의미 있는가? 환자 안전은?" |
| 환자 | "내가 아플 때 이걸 진짜 쓸까?" |
| 의료 전문가 | "FDA / HIPAA / 의료기기 분류 어디 해당?" |
| 보안 검수자 | "PHI 노출 위험은? 의료 데이터 컴플라이언스?" |

### 예시 2 — 이커머스
| 페르소나 | 핵심 질문 |
|---|---|
| 머천트 | "내 상품 등록/관리가 쉬워지는가?" |
| 구매자 | "이걸로 결제까지 무난한가? 신뢰는?" |
| 백엔드 전문가 | "재고/결제/세금 처리 정합성은?" |
| UX 검수자 | "전환 funnel 어디서 이탈할까?" |

### 예시 3 — 보안
| 페르소나 | 핵심 질문 |
|---|---|
| 침투 테스터 | "이걸 어떻게 깰 수 있을까?" |
| 관리자 | "운영하면서 어떻게 관리할까?" |
| 보안 아키텍트 | "공격 표면 / 신뢰 경계는?" |
| 규제 컴플라이언스 | "GDPR / SOC2 / ISO27001 충족?" |

### 예시 4 — 데이터 변환/분석
| 페르소나 | 핵심 질문 |
|---|---|
| 데이터 분석가 | "이 변환이 통계적으로 옳은가?" |
| 비개발자 사용자 | "csv 던져서 결과 받을 수 있나?" |
| 엔지니어 | "스키마/타입/null 처리 정합성은?" |
| 정합성 검수자 | "원본과 변환 결과 매핑이 정확한가?" |

### 예시 5 — 교육
| 페르소나 | 핵심 질문 |
|---|---|
| 교사 | "수업에서 어떻게 활용하지?" |
| 학생 | "내가 진짜 배울 수 있나?" |
| 교육과정 전문가 | "어떤 학년/단계에 맞나?" |
| 학습 효과 검수자 | "학습 결과 측정 가능한가?" |

### 예시 6 — 콘텐츠/미디어
| 페르소나 | 핵심 질문 |
|---|---|
| 크리에이터 | "내 작업 흐름에 fit 하는가?" |
| 시청자/독자 | "이 콘텐츠를 받았을 때 느낌?" |
| 편집자 | "퀄리티 게이트는 어디?" |
| 저작권 검수자 | "라이선스/저작권 침해 위험은?" |

→ 위 예시들은 **메뉴가 아니다**. 같은 헬스케어라도 "환자 앱" / "의료진 일정" / "임상시험 매칭"마다 4명이 다르다. 항상 아이디어를 보고 즉석 정의.

## 도메인 추정이 모호한 경우

명확하지 않으면 §A 블록으로 인터뷰한다:

```text
이 아이디어는 [추정 도메인]에 가까워 보이는데 맞나요?
4 관점에서 검증할 4명을 함께 정의할게요 (인원수 4 유지):
1. [방향성/비전] 관점은 누가? 2. [실제 사용]은? 3. [도메인 전문성]은? 4. [검증/엣지]는?
모르면 예시 0(기본)으로 진행하겠습니다.
```

## 절대 규칙

1. **인원수 4 유지** (검증 다중성 본질).
2. **4명은 항상 다른 4 차원** (방향성 / 사용 / 전문성 / 검증).
3. **본 문서를 메뉴로 사용 금지** — 예시는 패턴 학습용.
4. **도메인 추정이 모호하면 예시 0 사용** — 잘못된 매핑보다 안전.
5. **사용자 confirm 후 진행** — silent execution 금지.
6. **새 도메인 사례를 본 문서에 추가하지 말 것** — enum 카탈로그가 아닌 패턴 예시집.
